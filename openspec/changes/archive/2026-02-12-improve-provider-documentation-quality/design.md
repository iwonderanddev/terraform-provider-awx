# Design: Improve Provider Documentation Quality

## Context

Documentation is generated by `cmd/awxgen` from manifest JSON and
OpenAPI-derived field metadata. Current output quality issues are systemic:

- Field descriptions often fall back to the generic placeholder
  `Managed field from AWX OpenAPI schema` when OpenAPI text is weak.
- Resource and data source pages usually include only one minimal example,
  which is not enough for reference-heavy or workflow-oriented AWX objects.
- Resource docs render `Argument qualifiers used below` in a way that is easy
  to confuse with argument list content.
- Existing docs validation (`docs-validate`) checks for section presence but
  does not enforce description quality, example depth, or qualifier placement.

Confirmed constraints for this change:

- Priority resources are `awx_job_template`, `awx_workflow_job_template`,
  `awx_project`, `awx_credential`, `awx_inventory`, and
  `awx_inventory_source`.
- `## Example Usage` may include up to 3 examples when needed.
- AWX concept primers are added only for complex resources.
- References use official AWX docs as the behavior source.
- Official AWX links in generated pages must be resource-specific links to the
  exact AWX concept page (or closest official section for that concept), not
  only the generic AWX documentation index page.
- Curated descriptions/examples are validated against live official AWX 24.6.1
  docs before finalizing output.
- Prioritized-resource curation metadata includes official AWX link provenance
  plus verification date.
- Enhanced pages include a dedicated `## Further Reading` section.
- Strict quality gates are enforced first on the prioritized resources.

## Goals / Non-Goals

**Goals:**

- Replace placeholder-style field descriptions with clear, user-oriented
  descriptions for prioritized resources first.
- Provide practical, runnable examples with required supporting
  resources/data sources, up to 3 examples in `## Example Usage`.
- Add concise AWX concept primers on complex resources only, with links for
  deeper reading.
- Fix qualifier rendering so qualifier guidance is not mixed into argument
  lists.
- Align generated docs with Terraform Plugin Framework guidance and AWX-focused
  clarity.
- Add automated checks that protect phase-1 quality standards.

**Non-Goals:**

- Recreate full AWX product documentation inside this provider.
- Change provider runtime behavior, API contracts, or import ID semantics.
- Apply strict quality gating to every generated page in phase 1.
- Create exhaustive example matrices that make docs harder to read.

## Decisions

### 1. Use a phased rollout with an explicit first-priority resource set

Reasoning:

- The quality gap is broad; a targeted first pass gives predictable scope and
  faster iteration.

Decision:

- Phase 1 applies full quality improvements and validation to six resources:
  - `awx_job_template`
  - `awx_workflow_job_template`
  - `awx_project`
  - `awx_credential`
  - `awx_inventory`
  - `awx_inventory_source`
- Phase 2 expands strict quality gates to all remaining resources and data
  sources in a single batch.

Alternative considered:

- Expanding phase-2 gates by smaller capability batches was rejected in favor
  of one all-at-once pass.

### 2. Use docs-specific curated metadata

Reasoning:

- `internal/manifest/field_overrides.json` is runtime/schema-focused and
  already dense.

Decision:

- Add a docs-enrichment source for page intros, field description overrides,
  curated links, complexity flags, and additional examples.
- Record per-prioritized-resource curation provenance in docs metadata using an
  official AWX link and verification date.
- Consume this source during `awxgen docs` generation only.

Alternative considered:

- Embedding docs behavior directly in runtime overrides was rejected due to
  mixed concerns and lower maintainability.

### 3. Use deterministic description resolution

Reasoning:

- Quality requires a strict description hierarchy instead of ad hoc fallback
  text.

Decision:

- Description precedence:
  1. Curated docs override.
  2. Existing manifest/OpenAPI description.
  3. Type/reference-aware fallback with meaningful Terraform/AWX wording.
- Remove and forbid the fallback sentence
  `Managed field from AWX OpenAPI schema` for phase-1 pages.

Alternative considered:

- OpenAPI-only descriptions were rejected because schema text is inconsistent
  for user-facing clarity.

### 4. Use a bounded multi-example strategy

Reasoning:

- Many resources need more than one usage path, but too many examples reduce
  readability.

Decision:

- Always include one base example in `## Example Usage`.
- Add 1-2 additional examples only for distinct high-value cases.
- Apply the max-3 rule only to `## Example Usage`; `## Import` is excluded.
- Keep examples composable and avoid unnecessary scaffolding.

Alternative considered:

- Single-example-only docs were rejected as too shallow.
- Unlimited example count was rejected as too noisy.

### 5. Add AWX concept primers only on complex resources

Reasoning:

- Primers help when AWX concepts are non-obvious, but are unnecessary overhead
  on simple resources.

Decision:

- Determine complexity from objective signals:
  - official AWX documentation complexity
  - number of resource parameters
  - number of interactions/dependencies with other resources
- Persist the complexity result as a per-resource flag in docs metadata.
- For complex resources, include a short concept paragraph plus links.

Alternative considered:

- Primers on every page were rejected as repetitive and verbose.

### 6. Use an AWX-only official-link policy

Reasoning:

- AWX links ensure semantic correctness and keep generated pages focused on the
  platform behavior being managed.

Decision:

- For enhanced pages, include:
  - resource-specific official AWX docs links for behavior/domain details
- Add a dedicated `## Further Reading` section for these links.
- Maintain an explicit object-to-official-AWX-doc mapping in generator logic,
  with fallback to closest canonical official AWX section when no exact page
  exists.

Alternative considered:

- Including non-AWX style-benchmark links was rejected to reduce noise and
  preserve a single source of behavior truth.

### 7. Restructure schema sections and fix qualifier placement

Reasoning:

- The current qualifier phrase can be mistaken as part of argument entries.

Decision:

- Move to clearer schema sections (`Required`, `Optional`, `Read-Only`).
- Keep qualifier guidance in a dedicated subsection when needed.
- Ensure qualifier guidance is never emitted as argument bullets.

Alternative considered:

- Keeping the current mixed argument format was rejected due to readability
  issues.

### 8. Enforce validation gates for priority resources first

Reasoning:

- Strict checks across all pages immediately would slow delivery.

Decision:

- In phase 1, enforce these rules on the six prioritized pages:
  - no generic placeholder description text
  - expected schema section shape
  - qualifier guidance correctly separated
  - example count between 1 and 3
  - official AWX links are resource-specific and are not generic index links
  - `## Further Reading` includes official AWX links only
  - concept-and-link requirements when page is flagged complex
  - prioritized docs metadata includes curation provenance (`officialAwxUrl`
    and `verifiedOn`)
- In phase 2, extend the same gates to all remaining resources and data
  sources in one batch.

Alternative considered:

- Manual review only was rejected as not scalable.

## Risks / Trade-offs

- [Risk] New docs metadata introduces maintenance overhead.
  Mitigation: keep schema simple, validate format, and limit phase-1 scope.
- [Risk] Example quality can drift over time.
  Mitigation: enforce per-page example and section checks in lint/tests.
- [Risk] Complexity flags for concept primers may become inconsistent.
  Mitigation: document objective criteria and review them in doc changes.
- [Risk] AWX wording may evolve relative to curated descriptions.
  Mitigation: include official AWX links and review during schema refreshes.

## Migration Plan

1. Finalize capability deltas in specs.
2. Collect and verify official AWX 24.6.1 resource pages online for mapped
   concepts.
3. Add docs-enrichment metadata model, including curation provenance fields,
   and loader in `cmd/awxgen`.
4. Implement rendering updates for descriptions, schema sections, qualifiers,
   primers, and resource-specific links.
5. Add curated content and up-to-3 examples for the six prioritized resources.
6. Extend `docs-validate` and generator tests for phase-1 quality gates,
   including resource-specific AWX link enforcement.
7. Extend the same gates to all remaining resources/data sources in one batch.
8. Regenerate docs and run `make docs`, `make docs-validate`, and `make test`.
